# Практическая работа 2: Модульное приложение для учета автозапчастей на C#

## Обзор

Данный проект демонстрирует создание модульного приложения на C# с использованием принципов внедрения зависимостей (Dependency Injection) и динамической загрузки модулей. Приложение спроектировано таким образом, что его функциональность может быть расширена без модификации основного ядра, путем добавления новых модулей.

Текущая реализация представляет собой прототип системы для **учета автозапчастей**, сфокусированный на складском учете и операциях приема/отгрузки.

Структура проекта:

*   **`src/Pr2.ModulesAndDi/`**: Основное ядро приложения, модели данных, сервисы и демонстрационные модули.
*   **`tests/Pr2.ModulesAndDi.Tests/`**: Юнит-тесты для ключевой логики приложения.

## Архитектура

Архитектура приложения состоит из двух основных частей: **ядра** и **модулей**.

### Ядро (Core)

Ядро (каталог `src/Pr2.ModulesAndDi/Core/`) отвечает за:

*   **Определение контракта модуля (`IAppModule`)**: Интерфейс, которому должны соответствовать все модули. Он включает:
    *   `Name`: Уникальное имя модуля.
    *   `Requires`: Список имен модулей, от которых зависит данный модуль.
    *   `RegisterServices(IServiceCollection services)`: Метод для регистрации служб модуля в контейнере внедрения зависимостей.
    *   `InitializeAsync(IServiceProvider serviceProvider, CancellationToken cancellationToken)`: Асинхронный метод для инициализации модуля и выполнения его основных действий.
*   **Определение контракта действия (`IAppAction`)**: Интерфейс для действий, которые могут быть выполнены модулями.
*   **Модели данных для предметной области**: 
    *   `Part.cs`: Модель для автозапчасти (ID, артикул, название, описание, цена).
    *   `StockItem.cs`: Модель для элемента склада (ID запчасти, количество, расположение).
*   **Интерфейсы репозиториев**: 
    *   `IPartRepository.cs`: Интерфейс для управления данными о запчастях.
    *   `IStockRepository.cs`: Интерфейс для управления данными о наличии запчастей на складе.
*   **Загрузка и упорядочивание модулей (`ModuleCatalog`)**: Статический класс, который:
    *   Обнаруживает модули, реализующие `IAppModule`, из сборок.
    *   Строит корректный порядок запуска модулей с учетом их зависимостей (используя алгоритм топологической сортировки Кана).
    *   Обрабатывает ошибки:
        *   `ModuleLoadException`: Выбрасывается, если требуемый модуль не найден или обнаружена циклическая зависимость.
*   **Точка входа (`Program.cs`)**: Инициализирует конфигурацию, загружает и запускает модули в правильном порядке, управляет DI-контейнером и выполняет действия, предоставляемые модулями.

### Модули (Modules)

Модули (каталог `src/Pr2.ModulesAndDi/Modules/`) представляют собой расширения функциональности приложения. Каждый модуль:

*   Реализует интерфейс `IAppModule`.
*   Регистрирует свои службы в DI-контейнере через метод `RegisterServices`.
*   Может выполнять определенные действия при инициализации через метод `InitializeAsync` или регистрировать `IAppAction` для последующего выполнения ядром.
*   **Ключевой принцип**: Ядро приложения ничего не знает о конкретной реализации модулей, оно взаимодействует с ними исключительно через контракт `IAppModule`. Это позволяет добавлять, удалять или заменять модули без изменения логики ядра.

### Внедрение Зависимостей (Dependency Injection)

Приложение активно использует контейнер внедрения зависимостей `Microsoft.Extensions.DependencyInjection`. Это обеспечивает слабую связанность между компонентами, упрощает тестирование и повышает гибкость системы.

*   Модули регистрируют свои службы в контейнере.
*   Компоненты (включая другие модули или действия) запрашивают необходимые им зависимости через конструкторы, а контейнер автоматически предоставляет их экземпляры.

## Демонстрационные модули

В проекте реализованы следующие демонстрационные модули (в `src/Pr2.ModulesAndDi/Modules/`), адаптированные для системы учета автозапчастей:

*   **`CoreModule`**: Базовый модуль, который предоставляет основные сервисы, такие как `IClock`, `IStorage` (если еще используется), а также новые репозитории `IPartRepository` и `IStockRepository`. Не имеет зависимостей.
*   **`LoggingModule`**: Добавляет функциональность логирования событий, специфичных для системы учета автозапчастей, через сервис `IAppLogger`. Зависит от `CoreModule`.
*   **`ValidationModule`**: Реализует логику валидации данных автозапчастей, например, проверку уникальности артикула. Использует `IPartRepository` и `IAppLogger`. Зависит от `CoreModule`.
*   **`ExportModule`**: Экспортирует данные о запчастях и их наличии на складе в CSV-файл (`stock_export.csv`). Использует `IPartRepository`, `IStockRepository` и `IAppLogger`. Зависит от `CoreModule` и `ValidationModule`.
*   **`ReportModule`**: Формирует отчет по остаткам на складе, выводя его в консоль. Использует `IClock`, `IPartRepository`, `IStockRepository` и `IAppLogger`. Зависит от `CoreModule` и `ExportModule`.

### Демонстрационные сценарии

Вы можете управлять порядком и набором запускаемых модулей, редактируя файл `src/Pr2.ModulesAndDi/appsettings.json`.

**Пример `appsettings.json`:**

```json
{
  "Modules": [
    "Core",
    "Logging",
    "Validation",
    "Export",
    "Report"
  ]
}
```

Чтобы увидеть различные сценарии, попробуйте следующее:

1.  **Нормальный запуск:** Оставьте `appsettings.json` как есть. Приложение должно успешно запуститься.
    *   `ValidationModule` добавит тестовую запчасть (если ее нет).
    *   `LoggingModule` будет регистрировать события.
    *   `ExportModule` создаст файл `stock_export.csv` с данными склада.
    *   `ReportModule` выведет отчет по складу в консоль.
2.  **Проверка уникальности артикула (ValidationModule):** Запустите приложение несколько раз. `ValidationModule` попытается добавить запчасть с уже существующим артикулом и `IAppLogger` зарегистрирует ошибку валидации. Чтобы увидеть успешное добавление, удалите `stock_export.csv` перед запуском, так как данные в `InMemoryPartRepository` будут очищены при каждом новом запуске приложения.
3.  **Отсутствующий модуль:** Удалите "Core" из списка `Modules` в `appsettings.json`. Приложение должно завершиться с ошибкой, указывающей на отсутствие модуля "Core", так как другие модули зависят от него.
4.  **Циклическая зависимость:** Измените зависимости одного из модулей (например, в `ExportModule.cs` сделайте так, чтобы он зависел от `ReportModule`, а в `ReportModule.cs` — от `ExportModule`). Приложение должно обнаружить циклическую зависимость и завершиться с ошибкой.

## Запуск приложения

Чтобы запустить C# приложение:

1.  Перейдите в директорию `src/Pr2.ModulesAndDi/`.
2.  Откройте терминал и выполните команду:
    ```bash
    dotnet run
    ```

## Запуск тестов

Юнит-тесты находятся в проекте `tests/Pr2.ModulesAndDi.Tests/`. Чтобы запустить тесты:

1.  Перейдите в директорию `tests/Pr2.ModulesAndDi.Tests/`.
2.  Откройте терминал и выполните команду:
    ```bash
    dotnet test
    ```

Все тесты должны пройти успешно, подтверждая корректность логики `ModuleCatalog` и работу внедрения зависимостей.
